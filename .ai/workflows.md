# Development Workflows & Observability Runbook

This document outlines developer routines, build instructions, CLI commands, background job management, database migrations, and security operations in **Artha**.

---

## 🛠️ 1. Development Environment Setup

### Prerequisites

Ensure your local host machine has these runtimes installed:

- **Python 3.12+** with the [uv](https://github.com/astral-sh/uv) package manager
- **Bun 1.0+** for the frontend application
- **Docker & Docker Compose** for local service dependencies
- **Ollama** running locally on the host machine (`ollama serve`)

---

## 💻 2. Command Reference

### Backend Workflows (Python & uv)

Before running the backend, navigate to `/backend` directory.

1. **Install and Sync Dependencies**:
   ```bash
   uv sync
   ```
2. **Apply Database Migrations**:
   ```bash
   uv run alembic upgrade head
   ```
3. **Start the API Server**:
   ```bash
   uv run uvicorn src.main:app --host 0.0.0.0 --port 8000 --reload
   ```
4. **Start the Arq worker (Background job pool)**:
   ```bash
   uv run arq src.workers.arq_worker.WorkerSettings
   ```
5. **Run Lint Checks**:
   ```bash
   uv run ruff check .
   ```
6. **Run the Test Suite**:
   ```bash
   uv run pytest
   ```

### Frontend Workflows (Next.js & Bun)

Before running the frontend, navigate to `/frontend` directory.

1. **Install Node modules**:
   ```bash
   bun install
   ```
2. **Start the Next.js Development Server**:
   ```bash
   bun run dev
   ```
3. **Run Type-checking Validation**:
   ```bash
   bun run typecheck
   ```
4. **Run Linter checks**:
   ```bash
   bun run lint
   ```
5. **Build for Production**:
   ```bash
   bun run build
   ```
6. **Complete Validation Check (Types + Lint + Build)**:
   ```bash
   bun run check
   ```

## 🗄️ 3. Database Migrations Lifecycle (Alembic)

All structural PostgreSQL modifications must go through Alembic migrations.

### Generating a New Migration

1. Set up or edit SQLAlchemy database models inside `backend/src/domain/models.py`.
2. Generate an autogenerated revision revision:
   ```bash
   uv run alembic revision --autogenerate -m "add_table_description"
   ```
3. **Review the autogenerated file** under `backend/alembic/versions/` to verify correctness.

### Applying & Rolling Back Migrations

- **Upgrade to latest database schema version**:
  ```bash
   uv run alembic upgrade head
  ```
- **Downgrade schema version by one unit step**:
  ```bash
   uv run alembic downgrade -1
  ```

## ⚡ 4. Background Job Processing (Arq & Redis)

We offload high-computation actions (ingestion, document parsing, embeddings) to background workers using Redis and the Arq task scheduler.

```
FastAPI Web App (API Router)
       │
       │ (enqueue_job "process_document" payload)
       ▼
  Redis Queue
       │
       │ (delivers job to worker)
       ▼
  Arq worker (arq_worker.py)
       │ (runs parse -> chunk -> vector loop)
       ▼
  Database State (status='completed')
```

### Ingestion Pipeline Operations

1. Document upload triggers `process_document(document_id: UUID)` worker job.
2. The worker updates the Document status in PostgreSQL from `pending` to `processing`.
3. Calls the document ingestion services to extract raw string buffers, split chunks, extract embeddings, and save the entities.
4. Updates document status to `completed` or `failed` based on processing results.

## ⚙️ 5. Configuration & Environment Variables

Create `.env` inside the workspace root (based on `.env.example` template):

| Variable Name          | Purpose                        | Example Value                                          |
| :--------------------- | :----------------------------- | :----------------------------------------------------- |
| `DATABASE_URL`         | PostgreSQL connection string   | `postgresql+asyncpg://postgres:pass@localhost:5432/db` |
| `REDIS_URL`            | Redis server target            | `redis://localhost:6379/0`                             |
| `OLLAMA_BASE_URL`      | Host ollama server endpoint    | `http://localhost:11434`                               |
| `OLLAMA_MODEL_PLANNER` | Local LLM script planner model | `gemma4:e4b`                                           |
| `OLLAMA_MODEL_EMBED`   | Local Vector Embedding model   | `nomic-embed-text`                                     |

## 🛡️ 6. Security Operations

1. **Password Hashing**: Protect credentials using modern PBKDF2 hashing routines equipped with unique salt configurations inside `auth/security`. Never save passwords in raw formats.
2. **Authentication tokens**: Issue JWT (JSON Web Tokens) with defined expiration settings, verified during every backend request handler trigger via the dependency lifecycle.
3. **SQL Injection Prevention**: Write all operations using SQLAlchemy ORM expressions or async query param maps. Never inject raw parameters manually into raw text query statements.
4. **Endpoint Rate Limiting**: Limit API routes to mitigate bot spamming (e.g., auth registration routes capped at 5 requests/minute).

## 📊 7. Observability & Observability Metrics

- **Langfuse Integration**: Map LLM agent pipelines using Langfuse to log complete reasoning graphs, token counts, retrieval confidence metrics, and query execution times.
- **Structured JSON Logging**: Keep production outputs structured inside standard stdout channels, avoiding verbose debug configurations.
